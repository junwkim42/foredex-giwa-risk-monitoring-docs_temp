# Korean–English Terminology

This glossary is excluded from GitBook navigation and controls both variants.

| 한국어 기준 | English standard | Note |
|---|---|---|
| GIWA Sepolia 테스트넷 | GIWA Sepolia testnet | Never describe as mainnet |
| 온체인 이상 징후 탐지 | on-chain anomaly detection | Keep `on-chain` hyphenated |
| 온체인 리스크 모니터링 | on-chain risk monitoring | Product-category wording |
| 의도적으로 설계한 데모 시나리오 | purpose-built demo scenario | Distinguish from a real malicious project |
| 데모 공시 공급량 | demo-disclosed supply | Avoid `official supply` |
| 최초 민팅량 | initial minted supply | Tie to the deployment state |
| 공급량 불일치 | supply discrepancy | Difference between the registered reference and observed on-chain value |
| 추가 민팅 권한 | additional minting capability | Do not use `backdoor` without evidence |
| 비정상 민팅 | anomalous minting | Does not establish illegality |
| 프로젝트 지갑 | project-controlled wallet | Use consistently after first definition |
| 대규모 순유출 | large net outflow | Include time window and threshold when cited |
| 저유동성 풀 | low-liquidity pool | Keep hyphenated |
| DEX 집중 매도 | concentrated DEX sell-off | Use `DEX Dump` only when referring to the UI rule label |
| 탐지 규칙 | detection rule | Do not call it a model unless verified |
| 예시 임계값 | illustrative threshold | Not a universal risk standard |
| 경고 / 심각 | WARNING / CRITICAL | Severity labels, not legal findings |
| 주기적 모니터링 | periodic monitoring | Avoid `real-time` until measured |
| 데이터 갱신 주기 | refresh interval | Distinct from detection latency |
| 탐지 지연 | detection latency | Publish as an observed value, not a guarantee |
| 오탐 / 미탐 | false positive / false negative | Both remain possible |
| 온체인 검증 | on-chain verification | Link to Explorer evidence |
| 체인 재구성 | chain reorganization (reorg) | Include in testnet limitations |
| 거래소 관점의 프로젝트 감독 | exchange-side project oversight | Do not imply adoption by an exchange |
| 더 신뢰할 수 있는 생태계 | a more trustworthy ecosystem | Avoid absolute safety guarantees |
| AI 기반 주소 판별 | AI-based address classification | ForeDex capability; include in the demo architecture only if verified |

English wording separates evidence, design intent, and future scope:

- Verified evidence: `detected`, `recorded`, `verified`
- General function: `monitors for`, `is designed to detect`
- Future scope: `planned`, `potential`, `may support`

Avoid conversational filler and the expressions specified by ForeDex: `this means`, `this shows`, `this suggests`, `That's important`, and `Why it matters`.
